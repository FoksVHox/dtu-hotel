# Frontend Architecture

## Entry point

`resources/js/app.tsx` bootstraps Inertia with React. Pages are auto-resolved from `resources/js/pages/**/*.tsx` using Vite's `import.meta.glob`. The title formatter appends `- {APP_NAME}` to every page title.

## Page map

```
pages/
  welcome.tsx                   # Public landing page
  dashboard.tsx                 # Calendar + stat cards (auth + verified)
  bookings/
    index.tsx                   # Booking list + create dialog
  rooms/
    index.tsx                   # Room table with status badges
  maintenance/
    index.tsx                   # Maintenance overview
  settings/
    profile.tsx                 # Edit name/email, delete account
    password.tsx                # Change password
    appearance.tsx              # Light/dark/system theme
    two-factor.tsx              # Enable/disable 2FA, view recovery codes
  auth/
    login.tsx
    register.tsx
    forgot-password.tsx
    reset-password.tsx
    confirm-password.tsx
    verify-email.tsx
    two-factor-challenge.tsx
```

Each file under `pages/` maps 1:1 to a Laravel route. Inertia resolves the component by name — `Inertia::render('bookings/index')` loads `pages/bookings/index.tsx`.

## Layout system

```
layouts/
  app-layout.tsx       # Authenticated shell (sidebar + header + content)
  auth-layout.tsx      # Centered card shell for auth pages
  settings-layout.tsx  # Settings sidebar + content
```

Pages import their layout and render it as a wrapper. Auth pages use `auth-layout`, all app pages use `app-layout`.

## Component layers

```
components/
  ui/                  # Primitive Radix UI wrappers (Button, Dialog, Input, Select…)
  booking/             # Booking-specific: create dialog, date-time picker
  calendar/            # Calendar grid, week header, booking detail dialog, legend
  dashboard/           # Stat cards, skeletons for deferred props
  app-sidebar.tsx      # Navigation sidebar
  app-header.tsx
  app-content.tsx
  heading.tsx          # Reusable page heading
  rooms-table.tsx      # Rooms listing table
  room-status-badge.tsx
  spinner.tsx
  trend-indicator.tsx
  …
```

The `ui/` layer wraps Radix UI primitives with project-specific Tailwind styles. Do not add business logic there. Feature-specific components (booking, calendar, dashboard) go in named subdirectories.

## Wayfinder — route helpers

Wayfinder reads Laravel routes at build time and generates TypeScript functions. Never hardcode URLs in React components.

```typescript
// Generated automatically — do not edit
import { bookings } from '@/actions/BookingController';
import { guests } from '@/routes/guests';

// Use in components:
router.post(bookings.store());          // POST /bookings
fetch(guests.search({ q: term }));      // GET /guests/search?q=...
```

Files live in `resources/js/actions/` (controller-based) and `resources/js/routes/` (named routes). They are regenerated by `vite` at dev time and by `yarn build` at build time. **Never edit these files manually.**

The Vite plugin is configured with `formVariants: true` (`vite.config.ts:21`), which generates `.form()` variants suitable for `useForm` submissions.

## State management

There is no global state library. Data flows as:

1. **Inertia props** — server pushes data to the page on every navigation. Access via `usePage().props` or directly as component props typed by the page's `PageProps` interface.
2. **React local state** — `useState`/`useReducer` for ephemeral UI state (open/close dialogs, form field state before submission).
3. **Inertia `useForm`** — manages form field values, errors, and submission state for forms that POST back to Laravel.

There is no client-side cache layer. If you need fresh data, navigate to the page (Inertia re-fetches from server).

## TypeScript conventions

- All page props are typed via interfaces in `resources/js/types/`
- Strict mode is enabled in `tsconfig.json` — `any` is not permitted
- Named exports only — no default exports
- Wayfinder functions are the only way to reference backend routes

## Theme / appearance

The `HandleAppearance` middleware reads the `appearance` cookie (`light`|`dark`|`system`) on every request and sets it in the Inertia shared data. `initializeTheme()` in `app.tsx` applies the correct class to `<html>` before first render to avoid flash.
